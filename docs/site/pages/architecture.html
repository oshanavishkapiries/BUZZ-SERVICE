Architecture

Understand the technical architecture and design decisions behind Buzz Notification Service.

Overview

Buzz is designed as a highly scalable, multi-channel notification delivery platform built with modern cloud-native principles. The architecture follows a microservices-inspired approach with clear separation of concerns.

Design Principles
  • Horizontal scalability through stateless design
  • Queue-based asynchronous processing
  • Provider abstraction for flexibility
  • Comprehensive observability
  • Fault tolerance and retry mechanisms

System Architecture

The system is composed of several interconnected components working together to provide reliable notification delivery.

Core Components

1. HTTP API Server

Built with Fiber v2, a high-performance Go web framework. Handles incoming HTTP requests and provides RESTful endpoints for notification management.

  • Authentication and authorization
  • Request validation and sanitization
  • Rate limiting per API key
  • Health checks and readiness probes

2. Service Layer

Contains business logic for notification processing, template management, and datasource integration.

  • Notification Service: Creates, validates, and queues notifications
  • Template Service: Manages templates and variable substitution
  • Batch Service: Handles bulk notification operations
  • Datasource Service: Integrates with external data sources

3. Queue System

Uses Redis with Asynq for reliable message queuing and task scheduling.

  • Asynchronous task processing
  • Priority queues for urgent notifications
  • Automatic retry with exponential backoff
  • Dead letter queue for failed tasks
  • Scheduled/delayed notifications

4. Worker Pool

Concurrent workers process queued notifications and deliver them via appropriate providers.

  • Horizontal scalability (add more workers)
  • Channel-specific workers
  • Rate limiting per provider
  • Circuit breaker pattern for provider failures

5. Provider Abstraction

Each notification channel has multiple provider implementations with automatic fallback.

  • Email: Amazon SES (primary), SMTP (fallback)
  • SMS: NotifyLK (Sri Lanka), Twilio (international)
  • Push: Firebase Cloud Messaging
  • In-App: Server-Sent Events (SSE)

6. Data Layer

PostgreSQL 15+ with comprehensive schema design for notification tracking.

  • ACID compliance for data integrity
  • JSONB columns for flexible metadata
  • Full-text search capabilities
  • Partition-ready for scaling

Data Flow

Single Notification Flow

1. Client → POST /api/v1/notifications
           ↓
2. API validates request & API key
           ↓
3. Create notification record in DB (status: pending)
           ↓
4. Enqueue task to Redis queue
           ↓
5. Worker picks up task (status: processing)
           ↓
6. Worker calls provider (SES, Twilio, FCM, etc.)
           ↓
7. Update notification status (sent/failed)
           ↓
8. Return provider response to DB
           ↓
9. If failed: schedule retry with exponential backoff
           ↓
10. If in-app: push to SSE stream for real-time delivery

Bulk Notification Flow

1. Client → POST /api/v1/batches with datasource
           ↓
2. Create batch record (status: pending)
           ↓
3. Fetch recipients from datasource (Google Sheets, CSV, etc.)
           ↓
4. Apply template with variable substitution
           ↓
5. Create individual notification records for each recipient
           ↓
6. Enqueue all notifications to Redis
           ↓
7. Update batch counters (total, pending, success, failed)
           ↓
8. Workers process notifications in parallel
           ↓
9. Update batch status when complete

Notification Channels

Email Channel

  • Provider: Amazon SES (primary), SMTP (fallback)
  • Features: HTML templates, attachments, CC/BCC
  • Tracking: Opens, clicks (via SES SNS)
  • Rate Limit: Configured per SES quota

SMS Channel

  • Provider: NotifyLK (Sri Lanka), Twilio (global)
  • Features: Unicode support, sender ID customization
  • Tracking: Delivery receipts via webhooks
  • Rate Limit: Provider-specific

Push Notification Channel

  • Provider: Firebase Cloud Messaging (FCM)
  • Platforms: iOS, Android, Web
  • Features: Data payloads, rich media, badges
  • Device Management: Token registration and validation

In-App Channel

  • Protocol: Server-Sent Events (SSE)
  • Features: Real-time delivery, read/unread status, inbox
  • Persistence: Messages stored in inbox table
  • Scalability: Redis Pub/Sub for multi-instance support

Database Schema

The database is designed for scalability, flexibility, and comprehensive tracking.

Core Tables

  • notifications: Individual notification records with full lifecycle tracking
  • batches: Bulk notification batches with counters and status
  • templates: Reusable templates with variable substitution
  • inbox: In-app notification inbox with read/unread status
  • datasources: External data source configurations
  • api_keys: API authentication with rate limiting
  • device_tokens: Push notification device registrations

Key Features

  • Soft deletes on all tables
  • Automatic timestamps (created_at, updated_at)
  • JSONB for flexible metadata and configurations
  • Comprehensive indexes for query performance
  • Foreign key constraints for data integrity

Scalability

Horizontal Scaling

All components are designed to scale horizontally:

  • API Servers: Stateless design, add more instances behind load balancer
  • Workers: Add more worker processes to increase throughput
  • Database: Read replicas for query scaling, partitioning for data growth
  • Redis: Redis Cluster for high availability and scalability

Performance Optimizations

  • Connection pooling for database and Redis
  • Batch inserts for bulk operations
  • Indexes on frequently queried columns
  • Caching for templates and configurations
  • Goroutines for concurrent processing

Security

Authentication

  • API key-based authentication
  • Keys stored as bcrypt hashes
  • Environment-specific keys (production, staging, test)
  • Scoped permissions per API key

Rate Limiting

  • Per-minute, per-hour, per-day limits
  • Configurable per API key
  • Implemented using Redis sliding window
  • 429 responses when limits exceeded

Data Protection

  • HTTPS/TLS for all communications
  • Encrypted secrets in configuration
  • No sensitive data in logs
  • Soft deletes for data retention compliance

Network Security

  • IP whitelisting for API keys
  • Firewall rules for database and Redis
  • Private subnets for backend services
  • Network policies in Kubernetes
Production Ready

The architecture is designed with production workloads in mind, supporting millions of notifications per day with high availability and fault tolerance.
